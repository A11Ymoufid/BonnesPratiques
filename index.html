Checklist accessibilité

Liste de critères à vérifier pour améliorer l'accessibilité selon le RGAA 4.1.2

Structure et balisage général

  • La langue de la page est spécifiée sur la balise html avec l’attribut lang (ce même attribut est également utilisé à l’intérieur de la page sur les balises englobant des textes dans une autre langue afin de signaler un changement de langue)
    <html lang="fr-FR"> ... <a href="#">Notre sélection du <span lang="en-US">Black Friday</span></a> ... </html>
  • La balise title est correctement renseignée avec un intitulé pertinent qui reflete le contenu ou la fonction de la page
    Pour les pages d'archives WP la pagination est intégrée au title (ex : Tous les articles - Page 1 sur 3)
  • Le html est bien structuré avec des attributs role sur l'entête, les navigations, la zone de contenu principale, le footer et le moteur de recherche. Seul le rôle navigation peut être présent plusieurs fois dans la page, les autres sont uniques.
    Extension Landmarks pour tester la bonne implémentation.
    <header role="banner"></header> <nav role="navigation"></nav> <div role="search"> <form>...</form> </div> <main role="main"></main> <footer role="contentinfo"></footer>
  • Les titres de la page sont bien hiérarchisés :
    • un seul h1
    • bonne imbrication des h2, h3, etc : dans l'ordre et sans omission d'un niveau.
    Extension HeadingsMap (Chrome et Firefox) pour vérifier la bonne hiérarchisation.
  • Les attributs src des images sont tous remplis
  • Les balises a contiennent un texte permettant d’identifier le contenu de la page vers laquelle ils renvoient
    Pour en savoir plus sur nos CVG : <a href="/">cliquez ici</a> <a href="/">Consulter nos CGV</a>
  • Il n’est pas utilisé de balises standards (p, div, span) pour des éléments censés être structurés (listes, titres).
    <p class="h1">Mon titre</p> <h1 class="h1">Mon titre</h1>
  • Les pages du site sont conformes w3c
    Test de validité w3C : http://validator.w3.org/
  • Aucun élément de type p, a, li, button, legend, caption, figcaption, quote n'est vide.
    S’il y a besoin d’un bouton avec un icon uniquement, ajouter tout de même un texte à l’intérieur qui sera masqué à l’écran. Pour masquer ce type d'éléments ne pas utiliser d'astuces du type "display: none" ou "visibility: hidden" (qui empêchent la lecture par les lecteurs d'écrans). Exemple : https://gist.github.com/ffoodd/000b59f431e3e64e4ce1a24d5bb36034
    <button class="icon-loupe"><span class="hide-screens">Rechercher</span></button>
  • Si une description est associée à un bouton ou tout autre élément, elle doit être liée à l'élément en question via l'attribut aria-describedby
    <button aria-describedby="descriptionRetour">Retour</button> <div id="descriptionRetour">Le retour annulera toutes vos modifications non enregistrées. </div>

Formulaires

  • Les éléments de type form ont tous un bouton de validation (button ou input de type submit) avec un libellé pertinent
  • Les labels sont tous associés à un champ de formulaire (attribut for)
  • Les champs de formulaire ont tous un label associé :
    • Soit avec un id correspondant à l’attribut for d’un label <label for="recherche">Rechercher</label> <input type="search" id="recherche" />
    • Soit avec un attribut aria-label (si l’on souhaite se passer de label visible par exemple et utiliser un placeholder à la place) <input type="search" aria-label="Rechercher" placeholder="Rechercher" />

      aria-label peut être utilisé uniquement si une alternative visuelle comporte la même valeur (label ou placeholder)

    • Soit avec un attribut aria-labelledby correspondant à l'id du label <p id="label-recherche">Rechercher</p> <input type="search" aria-labelledby="label-recherche" />
    • Soit avec un attribut title <input type="search" title="Rechercher" />
  • Les listes de boutons radio et checkbox possédant un "sur-label" sont wrappées dans un fieldset avec un élément legend pertinent
    <fieldset> <legend>Civilité</legend> <input type="radio" name="civilite" id="civilite_monsieur" /> <label for="civilite_monsieur">Monsieur</label> <input type="radio" name="civilite" id="civilite_madame" /> <label for=" civilite_madame ">Madame</label> </fieldset>
  • Les autres types de champs nécessitant d'être groupés (par exemple 3 champs jour/mois/année) doivent également être wrappés dans un fieldsetavec un élément legend pertinent
  • Les champs obligatoires sont identifiables :
    • Soit avec une indication dans le label <label for="nom">Nom (champ obligatoire)</label> <input type="text" id="nom" />
    • Soit avec une aide visuelle et un texte explicatif situé avant le formulaire <p>Les champs obligatoires sont précédés d'une étoile *</p> <label for="nom">Nom *</label> <input type="text" id="nom" /> <label for="nom">Nom *</label> <input type="text" id="nom" /> <p>Les champs obligatoires sont précédés d'une étoile *</p>
    • Soit avec un attribut required ou aria-required doublé d'une aide visuelle (cf. point précédent)
  • Les formats attendus sont clairement précisés :
    • Soit dans le label <label for="date">Date (jj/mm/aaaa)</label> <input type="text" id="date" />
    • Soit dans un attribut title <label for="date">Date</label> <input type="text" id="date" title="jj/mm/aaaa" />
    • Soit à l'aide d'un attribut aria-labelledby ou aria-describedby <label for="date">Date</label> <input type="text" id="date" aria-describedby="help-format-date" /> <span id="help-format-date">jj/mm/aaaa</span> <label for="date">Date</label> <input type="text" id="date" /> <span>jj/mm/aaaa</span>

      aria-labelledby se substitue au label, alors que aria-describedby vient en complément. Dans les 2 cas ils doivent avoir comme valeur l'id de la balise contenant le texte qui doit être utilisé comme aide à la saisie.

  • Les erreurs de formulaire doivent être identifiées :
    • Soit dans le label des champs invalides
    • Soit avec un attribut aria-invalid dans les champs invalides
    • Soit via un attribut aria-describedby sur les champs invalides
    • Soit dans un texte situé avant les champs et listant toutes les erreurs

Tableaux

  • Les tableaux de données possèdent une légende pertinente (balise caption)
    <table> <caption>Tarifs 2019</caption> <tr>...</tr> </table>
  • Les entêtes de tableaux (colonnes et lignes) sont bien définies via des balises th
  • Les cellules de tableaux de données sont associées à des entêtes:
    • si les entêtes s'appliquent à toute la ligne ou colonne, les balises th possèdent un attribut scope="col" (entête de colonne) ou scope="row" (entête de ligne)
    • si les entêtes ne s'appliquent pas à toute la ligne ou colonne, les th possèdent un id et les td un attribut headers contenant la liste de toutes les entêtes auxquelles ils sont associés
  • Les tableaux de mise en forme possèdent un attribut role="presentation" et sont dépourvus de balises propres aux tableaux de données (caption, th, thead, etc), et d'attributs de type scope ou headers
    Un tableau de mise en forme est une balise table dont l'usage d'origine (l'affichage de données) est détourné en vue de mettre en page des éléments

    • Navigation

  • La page comporte au moins un lien d'évitement (lien d'accès rapide au contenu principal). Il est présent tout en haut du DOM. Il peut être présent en permanence ou être masqué à l'écran et affiché au focus pour les navigations au clavier.
    Pour masquer les liens d'évitement ne pas utiliser d'astuces du type "display: none" ou "visibility: hidden".
    Exemple : https://codepen.io/access42/pen/VVpWxJ
  • Une alternative non visuelle est proposée pour identifier la page active dans les menus :
    • avec un attribut title sur le lien de la page active <a href="/" title="Accueil - Page active">Accueil</a>
    • avec un texte masqué dans le libellé du lien de la page active <a href="/" title="Accueil">Accueil<span class="sr-only"> Page active</span></a>
      Attention aux propriétés utilisées pour cacher à l'écran seulement ("display:none" par exemple masque aussi l'élément sur les lecteurs d'écran. Exemple de css pour masquer à l'écran et conserver une visibilité par les lecteurs d'écrans : https://gist.github.com/ffoodd/000b59f431e3e64e4ce1a24d5bb36034)
  • Si l'outline d'un élément est supprimé, il est remplacé par un autre style permettant de visualiser le focus
  • Il est possible de naviguer sur le site uniquement avec le clavier : l'ordre des tabulations est cohérent et on peut toujours revenir en arrière ou avancer (pas de piège au clavier qui nous bloque).
    Tester en utilisant uniquement Tab pour avancer et Shift + Tab pour revenir en arrière.
    Outil pour mieux visualiser le focus : Bookmarlet (déplacer le lien dans la barre de favoris, se rendre sur le site à tester et cliquer sur le favoris pour lancer le js).
    Extension Accessibility Insights for Web option "Tab stops" pour visualiser le chemin parcouru avec les tabulations.
  • Aucune page publique du site n'est exclue du moteur de recherche

Apparence et images

  • Les contrastes fond/textes sont suffisants
    Pour faire une vérification rapide du contraste entre une couleur de fond et une couleur de texte : https://contrast-ratio.com/.
    À voir aussi l'application Colour Contrast Analyzer, le site Accessible Brand Colors (calcul des contrastes de toutes les couleurs d'une charte en même temps), le site Contrast Finder (propose des couleurs alternatives proches mais plus contrastées), l'outil AccessConfig (switcher à mettre sur les sites dont les couleurs ne sont pas assez contrastées)
  • Les éléments focus sont identifiables (si les styles par défaut ont été supprimés, ils doivent avoir été remplacés)
    input { outline: none; &:focus { border: 1px solid #000; } }
  • Tous les textes font partie du html et ne sont pas incorporés à des images
  • Les balises img ont un attribut alt contenant un texte pertinent
  • Les balises img décoratives ont un attribut alt vide
  • Les images svg décoratives ont un attribut aria-hidden="true"
  • Les légendes et leurs images associées sont wrappées dans une balise figure ayant un attribut role="group"
    <figure role="group"> <img src="/" alt="Chaton jouant dans son panier" /> <figcaption>Un adorable chaton</figcaption> </figure>
  • Le site n’utilise pas de captcha (ou une alternative, audio par exemple, est proposée)
  • Le site est utilisable sans images et background :
    • Les contrastes entre les textes et les couleurs de fonds remplaçant les background sont suffisants
  • Tous les textes possèdent une font-size de minimum 10px
  • Aucune couleur sur le site n'est porteuse à elle seule d'une information
    Ex : une bordure rouge sur un champ ne peut pas être le seul indicateur d'une erreur dans le formulaire, un message doit être ajouté
